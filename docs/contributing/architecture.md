# Architecture

How RDS 2.0 is put together вҖ” the moving parts you'll touch when contributing.

## Two-package model

```
@cuweb/rds-icons    (private, GitHub Packages вҖ” Carleton org only)
    вҶ‘ peerDependency
@cuweb/rds-2.0      (this repo)
    вҶ‘ dependency
Consumer project
```

Two packages, released independently. `rds-2.0` declares a version range for `rds-icons` in `peerDependencies`; consumers install both. See [maintenance/releasing.md](../maintenance/releasing.md) for the release flow that spans both repos.

## Two-package icon model

### Why two packages

Font Awesome Pro's [license](https://fontawesome.com/license) prohibits:

1. Distributing standalone copies of Pro Icons to non-Creators
2. Giving non-Creators permission to embed Pro Icons in their own projects

Carleton's FA Pro Organization license covers Carleton developers as "Creators," but **not** the general public who might `npm install` a public package.

If FA Pro SVGs were baked into `rds-2.0` and published publicly, anyone could install it and get the Pro Icons вҖ” a license violation. Same for serving the SVGs from a public CDN.

### The split

`rds-2.0` ships public, but contains **no Pro Icon data**. The icon data lives in a private companion package (`rds-icons`) published to GitHub Packages, which requires Carleton authentication to install. The GitHub Packages auth gate is what enforces license compliance вҖ” anyone can install `rds-2.0`, but the `<Icon>` component won't resolve without `rds-icons` present at install time.

### Key files in this repo

- [`src/components/Icon/Icon.tsx`](../../src/components/Icon/Icon.tsx) вҖ” the `<Icon>` component. Imports `iconMap` and `IconName` from the peer package.
- [`src/components/Icon/Icon.stories.tsx`](../../src/components/Icon/Icon.stories.tsx) вҖ” stories demonstrating color approaches and the full icon grid.

### Not in this repo (lives in `rds-icons`)

- FA Pro `.svg` source files
- The icon generator script
- Generated icon components
- `iconList` for Gutenberg

## Token pipeline

Design tokens are defined once in [`c2b.config.json`](../../c2b.config.json) and generated by [`@troychaplin/component2block`](https://www.npmjs.com/package/@troychaplin/component2block) into:

- `src/styles/auto/tokens.css` вҖ” CSS custom properties (`--rds--*`)
- `src/styles/auto/_variables.scss` вҖ” SCSS variables (for `@media` queries вҖ” CSS custom properties don't work there)
- `src/styles/auto/base-styles.{css,scss}` вҖ” generated element styles + layout classes
- `src/styles/auto/layout.css`, `typography.css` вҖ” generated stylesheets
- `dist/cutheme/theme-rds.json` and `dist/cutheme/integrate.php` вҖ” WordPress theme.json + integration helper

Run `pnpm c2b` to regenerate any of the above. Files in `src/styles/auto/` are never edited by hand.

The c2b tool itself lives in a separate repo вҖ” schema/generator changes happen there, not here.

## Build outputs

`pnpm build` runs three things in sequence:

1. **Asset generation** ([`scripts/build-assets.mjs`](../../scripts/build-assets.mjs)) вҖ” produces [`src/config/assets.ts`](../../src/config/assets.ts) and `src/styles/auto/_assets.scss` from [`src/config/assets.json`](../../src/config/assets.json).
2. **Vite library build** ([`vite.config.ts`](../../vite.config.ts)) вҖ” single-entry main bundle: `dist/index.{mjs,cjs,d.ts}` + `dist/style.css`.
3. **Vite per-component build** ([`vite.config.components.ts`](../../vite.config.components.ts)) вҖ” multi-entry build that emits `dist/components/<Name>/index.{mjs,cjs,d.ts}` for each name in the `componentNames` array.
4. **SCSS preprocess** ([`scripts/build-components-scss.mjs`](../../scripts/build-components-scss.mjs)) вҖ” emits `dist/components/<Name>/styles.{css,scss}` so consumers can `@import` per-component styles.
5. **`pnpm c2b`** вҖ” regenerates tokens + theme files (last step, so dist is up to date).

The `exports` map in [`package.json`](../../package.json) wires this layout to subpath imports. Adding a new component requires updating the `componentNames` array вҖ” see [conventions.md](conventions.md#adding-a-new-component).

## Storybook

Storybook 10 with `@storybook/react-vite`. Stories live next to components (`Foo.stories.tsx`) and are picked up by the glob in [`.storybook/main.ts`](../../.storybook/main.ts):

```ts
stories: ['../src/**/*.mdx', '../src/**/*.stories.@(js|jsx|mjs|ts|tsx)']
```

A11y testing runs through `@storybook/addon-a11y` + `@vitest/browser-playwright`. The `a11y.test: 'error'` parameter in [`.storybook/preview.ts`](../../.storybook/preview.ts) makes axe violations fail the build. See [testing.md](testing.md).

## Local development with rds-icons

When working on rds-2.0 itself, you need `rds-icons` checked out locally (it's a private peer dep). The dev convention is `~/Develop/personal/{rds-2.0,rds-icons}` вҖ” see [local-setup.md](local-setup.md) for the setup steps.

`rds-icons` points its `main`/`module`/`types`/`exports` at `src/` during local development (no build needed). On publish, `publishConfig` overrides those fields to point at `dist/`. This means changes to `rds-icons/src/` are picked up immediately by rds-2.0's Storybook вҖ” re-run `pnpm generate` in rds-icons and reload.
