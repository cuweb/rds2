# TODO 2 — Multi-output build, per-component CSS entries, and package rename

**Status:** Blocked on TODO 1 (needs at least one component to validate
per-component CSS emission).

## Context

The published package needs to serve two very different consumer
shapes:

1. **Modern frameworks (Next.js, Vite apps, etc.)**
   - Want tree-shakable JS and a single global stylesheet import.
   - `import { Button } from '@carletonuniversity/rds-2.0'`
   - `import '@carletonuniversity/rds-2.0/styles'`

2. **WordPress block plugins and block themes**
   - Want **global class names** (no CSS-module hashes) so they can be
     used in block render callbacks, theme.json, and server-side markup.
   - Need to import just one component's CSS from a block's
     `editor.scss`:
     ```scss
     @import '@carletonuniversity/rds-2.0/components/Button/style.css';
     ```
   - Accept that CSS will be duplicated across blocks that use the same
     component — that's an explicit trade-off, not a bug.

## Target `dist/` shape

```
dist/
  index.mjs                       // main ES entry, tree-shakable
  index.cjs                       // main CJS entry
  index.d.ts                      // main types
  style.css                       // tokens + globals + ALL components

  tokens.css                      // CSS custom properties only
  globals.css                     // normalize + base element styles (no components)

  components/
    Button/
      index.mjs                   // per-component ES entry
      index.cjs                   // per-component CJS entry
      index.d.ts                  // per-component types
      style.css                   // ONLY .rds-button* rules

  cutheme/                        // unchanged — generated by c2b
    theme-rds.json
    integrate.php
```

## Target `package.json` `exports`

```jsonc
{
  "name": "@carletonuniversity/rds-2.0",
  "exports": {
    ".": {
      "types": "./dist/index.d.ts",
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs"
    },
    "./styles": "./dist/style.css",
    "./tokens": "./dist/tokens.css",
    "./globals": "./dist/globals.css",
    "./components/Button": {
      "types": "./dist/components/Button/index.d.ts",
      "import": "./dist/components/Button/index.mjs",
      "require": "./dist/components/Button/index.cjs"
    },
    "./components/Button/style.css": "./dist/components/Button/style.css"
  }
}
```

Each new component added in future TODOs gets two additional entries
in this map (JS + CSS). Worth considering: a small script that
generates the `exports` block from the filesystem during `pnpm build`,
to avoid drift as components are added.

## Changes

### `package.json`
- Replace `exports` block with the target map above.
- Keep `files: ["dist"]`.
- Leave `size-limit` targets; revisit after first build to verify new
  file layout.

### `vite.config.ts`
- Switch from single `entry: 'src/index.ts'` to an entry map:
  ```ts
  build: {
    lib: {
      entry: {
        index: 'src/index.ts',
        'components/Button/index': 'src/components/Button/index.ts',
        'styles/tokens': 'src/entries/tokens.ts',
        'styles/globals': 'src/entries/globals.ts',
      },
      formats: ['es', 'cjs'],
      fileName: (format, name) =>
        `${name}.${format === 'es' ? 'mjs' : 'cjs'}`,
      cssFileName: 'style',
    },
    cssCodeSplit: true,
    rollupOptions: {
      external: ['react', 'react-dom', 'react/jsx-runtime'],
    },
  },
  ```

### New CSS-only entry wrappers
- `src/entries/tokens.ts` — only imports `../styles/auto/tokens.css`
- `src/entries/globals.ts` — only imports a new
  `../styles/globals-only.scss` file that `@use`s `modern-normalize`,
  `base/globals`, and `auto/base-styles` (no tokens, no components)

### New SCSS composition file
- `src/styles/globals-only.scss` — normalize + base + base-styles,
  no tokens. Rationale: `tokens.css` is its own entry, so `globals.css`
  should not duplicate the custom properties.

### Post-build cleanup (if needed)
- The `tokens` and `globals` entries will produce zero-content JS files
  alongside their CSS. Either:
  1. Leave them (tiny, harmless), or
  2. Add a post-build script to delete
     `dist/styles/tokens.{mjs,cjs}` and `dist/styles/globals.{mjs,cjs}`
     so consumers only see the CSS files.
  Choose based on what testing reveals.

## Build-pipeline risks

Vite library mode + multiple entries + `cssCodeSplit: true` is a less
common combination. Known risks:

1. Lib mode may concatenate all CSS into a single file regardless of
   `cssCodeSplit` (historical behavior).
2. CSS chunk filenames may not line up with the entry structure we want.
3. A CSS-only entry (no real JS imports) may emit an empty chunk or
   fail entirely.

**Fallback plan** if lib mode can't cleanly split CSS per entry:

- Keep the main lib build for JS + the bundled `dist/style.css`.
- Add a **second Vite build step** (separate config, invoked from
  `pnpm build`) that compiles each component's SCSS into a standalone
  CSS file under `dist/components/*/style.css` using a small script
  that walks `src/components/*/`.
- `tokens.css` and `globals.css` can be handled by a plain copy/SCSS
  compile step — no JS wrapper needed.

Decide between the single-config and two-build approaches during
implementation based on what Vite actually produces.

      except possibly empty `tokens.mjs`/`globals.mjs` stubs).

      base element rules, layout classes, `.rds-button*` rules.
